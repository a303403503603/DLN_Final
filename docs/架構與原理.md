## 2. 怎麼使用

### 2.1 每日自動化（主要用法）

```bash
# 盤中交易（10:00 執行）
python -m pipeline.run_daily_pipeline trade
python -m pipeline.run_daily_pipeline trade --simulate   # 模擬模式（不下真實單）

# 盤後更新（16:30 執行）
python -m pipeline.run_daily_pipeline daily              # fetch + gru + reconcile

# 單一階段
python -m pipeline.run_daily_pipeline fetch              # 只抓 API 新資料
python -m pipeline.run_daily_pipeline gru                # 只更新 GRU 快取
python -m pipeline.run_daily_pipeline full               # fetch + gru + trade
python -m pipeline.run_daily_pipeline status             # 看目前狀態
```

### 2.2 Linux 排程（cron）

```
# 週一~五 10:00 執行交易
0 10 * * 1-5 /path/to/run_pipeline.sh trade

# 週一~五 14:00 盤後更新
0 14 * * 1-5 /path/to/run_pipeline.sh daily
```

### 2.3 訓練

```bash
# GRU 預訓練（所有股票，約 100 epoch）
python -m training.train_gru

# PPO 訓練（2003~2022 訓練，2023~2024 驗證，約 200 episdes）
python -m training.train_ppo

# 超參數掃描
python -m training.train_ppo --max-cash-ratio 0.3 --excess-penalty 1.0 --ent-coef 0.01 --lr 3e-4 --label best

# 批量回測
python -m training.run_batch_sim
```

### 2.4 Discord Bot 控制

```bash
# 啟動 Discord bot（24/7 常駐）
python -m pipeline.discord_bot

# Windows 批次檔
run_bot.bat
```

Bot 啟動後會在你的 Discord 頻道傳送控制面板：

| 按鈕 | 用途 |

| 📊 Status | 查看現金、持股數、confirmed_trades |

## 3. 資料怎麼處理

### 3.1 資料來源

- **歷史資料**：`data/taiwan_stock_price.csv`，包含 1993~2026 年所有上市櫃股票日行情
- **API 資料**：每日透過 `get_taiwan_stock_data()` 從 TWSE 官方網站抓取當日即時資料
- **快取機制**：API 資料存到 `data/api_cache/{code}.pkl`，避免重複抓取

### 3.2 特徵工程（`data_processor.py`）

每檔股票每天產出 21 個數值特徵：

| 類別 | 特徵 | 數量 |
|------|------|------|
| 價格 | open, high, low, close, change | 5 |
| 成交量 | capacity, transaction_volume | 2 |
| 波動 | high_low_range, open_close_change | 2 |
| 技術指標 | ma_5/10/20/60/120_ratio, rsi_14 | 6 |
| 量能 | vol_ma_5_ratio, vol_ma_20_ratio | 2 |
| 動能 | momentum_5/10/20/60 | 4 |

### 3.3 資料流

```
CSV + API 合併資料
    │
    ▼ engineer_features() → 21 維特徵
    │
    ▼ StandardScaler 標準化（各股各自 fit，不跨股）
    │
    ▼ 滑窗 (5/10/20/60/120) → 5 個張量 → GRU forward
    │
    ▼ 輸出 .npy (天數, 1287)
    │  ┌ 1280: 多窗口 GRU 融合後的潛在向量
    │  └ 7:    未來 1/3/5/7/15/30/60 日預測報酬
    │
    ▼ PPO 環境查表讀取，每檔候選股取 1328 維
```

### 3.4 快取索引對齊

GRU 快取的天數長度 ≠ 資料天數長度（因滑窗需要前緣補齊）。對齊方式：

```python
offset = len(cache) - len(df)   # cache 尾部對齊 data 尾部
cache[day_idx]  →  對應 data 第 day_idx 天
```

此 `_get_cache_idx()` 邏輯在 `run_daily_pipeline.py` 和 `live_trading.py` 中都有實作。

## 4. 交易邏輯

### 4.1 每日交易步驟（`LiveTrader.run()`）

```
1. 檢查是否為交易日（週一~五 09:30~13:30）
2. 讀取投資組合狀態 (logs/portfolio_state.json)
3. 調 API Get_User_Stocks() 取得實際持股
4. 掛單對帳（處理 pending_buys / pending_sells）
5. 若仍有掛單未結清 → 跳過今日交易（防止重複下單）
6. 讀取 API 資料 + 特徵工程
7. GRU 快取對齊 + 篩選 30 檔候選股
8. 建構觀察向量（30檔 × 1328 維）
9. PPO 推理（stochastic=False，確定性模式）
10. 計算目標股數 + 限價
11. 執行 Sell_Stock / Buy_Stock API
12. 新訂單記錄為 pending_buys / pending_sells
13. 儲存狀態
```

### 4.2 掛單管理

```
from PPO decision → 下 API 單
       │
       ▼
記錄到 pending_buys / pending_sells
（此時不算 confirmed_trades，現金也暫不扣/加）
       │
       ▼ (隔日 run 時)
調 Get_User_Stocks() 對帳
       │
       ├── 股票出現在 API 持股 → buy confirmed!
       │    confirmed_trades += 張數
       │    從 pending_buys 移除
       │
       ├── 股票從 API 持股消失 → sell confirmed!
       │    confirmed_trades += 張數
       │    從 pending_sells 移除
       │
       ├── pending_buy 股票未出現 → 失效
       │    退款: cash += shares × 1000 × price × (1 + COMMISSION_BUY)
       │
       └── pending_sell 股票仍在 → 未成交
            現金不扣（當作沒賣成）
```

**重點：** `confirmed_trades` 才是真實成交的交易次數，也是判斷是否達到 100 筆目標的依據。

### 4.3 限價計算

```python
limit_price = close_price × (1 + gru_cache[day_idx, horizon_index_1])
limit_price = clip(limit_price, close × 0.95, close × 1.05)
```

使用 GRU 預測的 horizon-1（隔日報酬）來計算限價，而非直接用昨日收盤價。這是因為預測價格更接近市場預期，能提高成交機率。±5% 夾持防止極端預測。

### 4.4 現金估算法

API 沒有提供現金餘額，系統用成本基礎法推算：

```python
estimated_cash = INITIAL_CAPITAL
               - Σ(holdings[stock] × 1000 × cost_price)
```

如果推算值與既有狀態的現金差異超過 1%（100 萬元），就以推算值覆蓋。

## 5. PPO 環境設定

### 5.1 候選池篩選（`_screen_pool()`）

不固定股票池，每日動態更新：

1. 所有股票依 GRU t+1 預測報酬排序
2. 科技產業股（代碼 24,25,26,28,31）得分 × 1.15
3. 持倉的股票優先保留
4. 補最高分新股票到 30 檔
5. 不足則重複最後一檔補滿

### 5.2 動作空間

輸出 31 維權重向量：
- 前 30 維：各候選股目標權重
- 第 31 維：現金權重
- 使用 Dirichlet 分布確保正數且總和為 1

### 5.3 step 流程

```
收到 31 維權重
    │
    裁剪正值 → 歸一化
    │
    換手率限制 (≤ 20%)
    │
    現金約束 (5% ≤ cash ≤ 30%)
    │
    單股上限 (≤ 30%)
    │
    先賣（釋放現金）→ 不足時強制賣獲利股 → 停損
    │
    後買（受現金餘額限制）
    │
    池刷新（為下一日準備新候選股）
```

### 5.4 獎勵函數組成

```python
reward = r_profit       # 持股價格變動貢獻（profit_weight=10.0）
       + r_drawdown     # 回撤加深時懲罰（-0.5 × 增量）
       + r_turnover     # 換手率懲罰（-1.0）
       + r_invalid      # 無效動作懲罰（-0.01）
       + r_terminal     # 結束時總報酬獎勵（profit_weight × total_return）
       + r_excess_cash  # 持有過多現金懲罰（-1.0 × 超額量）
```

## 6. 訓練設定

### 6.1 GRU 訓練

| 項目 | 設定 |
|------|------|
| 資料範圍 | CSV 全部（1993~2026） |
| 時間窗口 | 5, 10, 20, 60, 120 天 |
| 預測標的 | 1/3/5/7/15/30/60 日後報酬 |
| Loss | 加權 Huber（近期權重 0.35 → 最遠 0.03） |
| 模型 | 3層 GRU, hidden=256, dropout=0.3, 共用編碼器 |
| 訓練 | Adam, lr=0.001, batch=2048, mixed precision |
| 早停 | validation loss 連續 10 epoch 未改善停止 |
| 產出 | checkpoints/gru_shared.pth |

### 6.2 PPO 訓練

| 項目 | 設定 |
|------|------|
| 訓練資料 | 2003~2022（top 100 交易量加權股） |
| 驗證資料 | 2023~2024 |
| 演算法 | PPO with GAE (γ=0.99, λ=0.95) |
| 超參數 | lr=3e-4, clip=0.2, ent_coef=0.01, vf_coef=0.5 |
| Episode | max 2048 steps |
| 策略分布 | Dirichlet（非 Gaussian） |
| 產出 | checkpoints/ppo_best.pth |

### 6.3 關鍵超參數（`config.py`）

```python
INITIAL_CAPITAL = 100_000_000    # 起始資金 1 億
CANDIDATE_POOL_SIZE = 30         # 每日候選股數
TURNOVER_LIMIT = 0.20            # 最高日換手率 20%
MAX_CASH_RATIO = 0.30            # 最高現金比 30%（防止不做單）
MIN_CASH_RATIO = 0.05            # 最低現金比 5%（預留保證金）
MAX_POSITION_RATIO = 0.30        # 單股上限 30%
STOP_LOSS_PCT = 0.15             # 停損 15%
COMMISSION_BUY = 0.001           # 買入手續費 0.1%
COMMISSION_SELL = 0.003          # 賣出手續費 0.3%
```

## 7. 踩過的坑 & 注意事項

### 7.1 GRU 快取索引不同步

GRU 因為滑窗需要前綴補 padding，`gru_cache/{code}.npy` 的天數會比資料 DataFrame 多 `MAX_W - 1` 行。環境中需要用 `offset = len(cache) - len(df)` 對齊尾部。沒對齊會讀到 2003 年的資料。

### 7.2 買入手續費沒被 import

`live_trading.py` 和 `run_daily_pipeline.py` 原本的 pending sell 退款邏輯中，`COMMISSION_BUY` 沒被 import 進來 → NameError。已在兩個檔案中補上 import。

### 7.3 現金 drfit 只在有持股時才修正

原版 drift 修正寫在 `if api_holdings:` 區塊內，如果完全空手就不會跑 → 現金會逐漸偏離正確值。已移出條件判斷，現在一定執行。

### 7.4 限價改為 GRU 預測

原本限價 = 昨日收盤價（容易因市價跳空而無法成交）。改為 `close × (1 + pred_h1)` 後，更接近市場預期。

### 7.5 MAX_CASH_RATIO 沒被套用

訓練環境 `trading_env.py:step()` 原本只有 MIN_CASH_RATIO 的限制，沒有 MAX 的上限 → 策略學到 100% 持有現金的退化行為。在 step 約 337~347 行補上 MAX_CASH_RATIO 的強制約束。

### 7.6 `Get_User_Stocks` 只有已成交持股

API 的 `Get_User_Stocks` 只回傳已搓合的持股，排隊中等著的訂單不會出現。所以只能用 `pending_buys/pending_sells` 機制來追蹤尚未確認的訂單。

## 8. API 說明

系統使用 NCKU 模擬交易 API（`https://ciot.imis.ncku.edu.tw/sim_stock/trading_api`）：

| 函式 | 用途 | 注意 |
|------|------|------|
| `Buy_Stock(account, pwd, code, shares, price)` | 下買單 | 回傳 success 不代表成交，只是進佇列 |
| `Sell_Stock(account, pwd, code, shares, price)` | 下賣單 | 同上 |
| `Get_User_Stocks(account, pwd)` | 查持股 | 只回已搓合的，回傳 list of dicts |
| `get_taiwan_stock_data(code, start, end)` | 抓歷史行情 | 交易時段內回傳空 DataFrame（TWSE 收盤後才公布） |

## 9. Discord Bot 控制系統

### 9.1 架構

```
┌──────────────┐     ┌─────────────────────┐
│  Discord App │◄───►│  discord_bot.py     │
│  (手機/PC)   │     │  (24/7 常駐行程)    │
└──────────────┘     │                     │
                     │  on_ready:          │
                     │   傳送控制面板       │
                     │                     │
                     │  check_trade_time:  │
                     │   週一~五 10:00     │
                     │   自動發起 trade    │
                     │                     │
                     │  do_trade_flow():   │
                     │   1. LiveTrader     │
                     │      compute_       │
                     │      proposal()     │
                     │   2. 傳送 Embed     │
                     │   3. 等待按鈕回應    │
                     │   4. Yes → 執行     │
                     │      Adjust → Modal │
                     │      No → 取消      │
                     └─────────────────────┘
```

### 9.2 按鈕面板

Bot 啟動時在 `DISCORD_CHANNEL_ID` 頻道送出一則 embed + 按鈕行（`ControlPanelView`）：

- **📊 Status**：呼叫 `Get_User_Stocks()` API，回傳現金估算值、持股數、confirmed_trades 數
- **📋 Holdings**：列出所有持股的代碼、張數、成本價、成本總額
- **💰 Price**：跳出 Modal，輸入股號（如 `2330`），回傳最近 10 日內的最新股價（注意：交易時段內 TWSE 無資料）
- **⚙️ Daily Update**：背景執行 `phase_fetch()` + `phase_gru()`（盤後更新用）
- **🔄 Trade Now**：執行完整 PPO 提案流程（等同 10:00 自動觸發）

### 9.3 Trade 提案流程

```
[使用者按 🔄 Trade Now 或 10:00 自動觸發]
    │
    ▼ 背景執行 compute_proposal()
        - 讀取 API 快取
        - Get_User_Stocks → 對帳
        - 現金 drift 修正
        - 篩選 30 檔候選股
        - PPO 推理 → 目標權重
        - 計算買賣張數、限價
    │
    ▼ 傳送 Embed + TradeProposalView
        - ✅ Yes, execute
        - ✏️ Adjust
        - ❌ No, cancel
    │
    ├── ✅ → asyncio.to_thread(execute_proposal)
    │         呼叫 Buy_Stock/Sell_Stock
    │         更新 pending_buys/pending_sells
    │         回報結果
    │
    ├── ✏️ → AdjustModal
    │         一行一檔，格式：代碼 張數 [@價格]
    │         範例: 2438 80 @26
    │               3041 150
    │               2392 0   (移除)
    │         送出後 → 重新計算金額 → 更新 Embed
    │
    └── ❌ → 取消，顯示 "Trade Cancelled"
```

先在 [Discord Developer Portal](https://discord.com/developers/applications) 建立 Application → Bot → 複製 Token。再用 OAuth2 URL Generator 邀請 bot 到伺服器。
